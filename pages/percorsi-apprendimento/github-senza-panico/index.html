Guida concreta orientata alla piattaforma

1. Cos'è GitHub in pratica

GitHub non è solo "dove fai push". È una piattaforma per gestire il ciclo completo del progetto: organizzazione del lavoro, collaborazione, review, automazioni e pubblicazione.

La differenza breve:

  • Git: gestisce la cronologia del codice.
  • GitHub: gestisce il progetto attorno al codice.

Questa guida è focalizzata proprio su questa seconda parte.

2. Account e sicurezza

Prima di tutto metti in sicurezza l'account, altrimenti rischi di perdere repository o esporre codice.

Checklist minima

  • Attiva 2FA (app autenticatore o passkey).
  • Genera codici di recupero e salvali offline.
  • Rivedi sessioni attive e dispositivi autorizzati.
  • Rimuovi token vecchi o non usati.

Se lavori da terminale usa token con permessi minimi, non più larghi del necessario.

3. Profilo GitHub e README profilo

Il profilo GitHub è la tua homepage tecnica. Curarlo bene aiuta molto più di avere tanti repo disordinati.

Cosa ottimizzare

  • Bio chiara in una riga.
  • Pin di 4-6 repository davvero rappresentativi.
  • README profilo con stack, focus e progetti principali.

Meglio pochi progetti ben spiegati che decine di repo vuoti.

4. Repository fatti bene

Quando crei un repository, imposta bene la base subito.

Struttura iniziale consigliata

  • README.md con obiettivo e istruzioni.
  • LICENSE se il progetto è pubblico.
  • .gitignore corretto per il tuo stack.
  • CONTRIBUTING.md se prevedi contributi esterni.

Pubblico o privato

Pubblico per portfolio o open source. Privato per lavori interni, test o materiale non pronto.

5. Collaboratori e permessi

Su GitHub i permessi contano tantissimo. Dare accesso "admin" a tutti è un errore classico.

Read
solo lettura.
Triage
gestione issue senza scrivere codice.
Write
push e gestione branch.
Maintain
gestione quasi completa del repo.
Admin
controllo totale, da limitare a pochi.

Usa il principio del minimo privilegio: ogni persona solo con i permessi che le servono davvero.

6. Issues, Labels e Projects

Le issue non sono solo bug list: sono il tuo sistema di lavoro.

Flusso semplice

  1. Crea issue piccole e chiare.
  2. Aggiungi label coerenti (bug, feature, docs, priority).
  3. Assegna owner e scadenza solo se utile.
  4. Collega le issue a una board Projects.

Con Projects puoi vedere subito cosa è backlog, in corso, in review e completato.

7. Pull Request workflow

La Pull Request è il cuore della collaborazione su GitHub.

Workflow consigliato

  1. Apri branch dedicato alla task.
  2. Push del branch e PR in stato Draft.
  3. Descrizione PR con obiettivo, scope e test.
  4. Richiedi review quando è pronta.
  5. Merge solo dopo check verdi e review chiusa.

Una PR piccola è più facile da capire, testare e approvare.

8. Code review utile

Su GitHub la review efficace non è "mi piace" o "ok". Deve lasciare valore tecnico.

Cosa controllare

  • Rischi di regressione.
  • Copertura test adeguata.
  • Naming e leggibilità.
  • Sicurezza base (input, secret, dipendenze).
  • Coerenza con architettura del progetto.

Commenti brevi, specifici e con proposta concreta sono i più utili.

9. Branch protection e regole

Se usi solo main senza protezioni, prima o poi qualcuno rompe la branch principale con un push diretto.

Impostazioni consigliate su branch principale

  • Blocca push diretto su main.
  • Richiedi almeno 1 approvazione review.
  • Richiedi status check passati prima del merge.
  • Richiedi branch aggiornata rispetto a main.

Con le rulesets centralizzi queste regole anche su più repository.

10. GitHub Actions

GitHub Actions automatizza build, test, lint e deploy ad ogni push o PR.

Pipeline minima utile

  • install dipendenze;
  • lint;
  • test;
  • build.

Se questi check sono obbligatori prima del merge, la qualità media del progetto sale subito.

11. GitHub Pages

Per siti statici, GitHub Pages è il modo più veloce per pubblicare una demo.

Quando usarlo

  • portfolio personale;
  • landing page di progetto;
  • documentazione tecnica.

Puoi pubblicare da branch dedicata o da pipeline Actions. Con dominio custom e HTTPS, hai setup completo senza infrastruttura complessa.

12. Release e changelog

Tag e release servono per dare versioni stabili e tracciabili al progetto.

Buona pratica

  • usa versionamento semantico (v1.2.0);
  • scrivi release notes chiare;
  • collega issue/PR principali della release.

Questo aiuta chi usa il progetto e aiuta anche te a capire cosa è cambiato versione per versione.

13. Sicurezza su GitHub

GitHub include strumenti molto utili che spesso vengono ignorati.

Strumenti da attivare

  • Dependabot alerts per vulnerabilità dipendenze.
  • Dependabot updates per aggiornamenti automatici.
  • Secret scanning per evitare token accidentalmente committati.
  • Code scanning (es. CodeQL) per analisi statica.

Se trovi un secret nel repository, ruotalo subito: non basta cancellarlo nel commit successivo.

14. Workflow consigliato

Da solo

  1. Issue rapida per ogni task non banale.
  2. Branch dedicato.
  3. PR anche senza team (ti aiuta a rileggere).
  4. Merge solo con check verdi.

In team

  1. Definition of done chiara in PR template.
  2. Review obbligatoria.
  3. Branch protection sempre attiva.
  4. Board Projects aggiornata in modo disciplinato.

15. Errori comuni su GitHub

  • usare main come sandbox senza branch;
  • aprire PR enormi con 50 cambi insieme;
  • lasciare repository pubblici senza README;
  • dare permessi admin a troppa gente;
  • ignorare alert di sicurezza;
  • usare Actions senza cache e senza controllo costi/tempi.

Correggere questi 6 punti cambia radicalmente la qualità del lavoro su GitHub.

16. Nota finale

Se vuoi usare GitHub bene, non basta "sapere fare push". Devi imparare a usare la piattaforma per organizzare, verificare e pubblicare in modo affidabile.

Parti da questi blocchi: repository puliti, PR disciplinate, review utili, Actions minime, regole di branch e sicurezza attiva.

Con questa base, GitHub smette di essere solo un archivio online e diventa davvero il tuo ambiente di sviluppo collaborativo.
